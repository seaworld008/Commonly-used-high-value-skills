# Traceability ‚Äî Canonical ID Scheme & `.traceability.yaml`

Shared source of truth for requirement/acceptance/test identifiers and the machine-readable
traceability ledger. Spec-authoring (`accord`, `scribe`) and spec-verification (`attest`) skills,
plus the review gate (`judge`, `guardian`), MUST use this scheme so links survive across skills
instead of each skill minting its own incompatible IDs.

Rationale: Spec-Driven Development requires intent (the spec) to remain the source of truth and
implementation to be a *derived, traceable* artifact. Unidirectional or per-skill IDs let
orphaned tests and untested requirements pass silently. One scheme + one ledger closes the loop.

---

## 1. Canonical ID Scheme

Every requirement, criterion, scenario, and downstream artifact uses one of these prefixes.
IDs are **stable** ‚Äî never renumber; supersede with `_v{N}` when a spec revision changes meaning.

| Prefix | Meaning | Owner skill | Format |
|--------|---------|-------------|--------|
| `REQ-{NNN}` | Functional / business requirement (user-facing need) | accord (`L1`), scribe (`FR`) | `REQ-001` |
| `CFR-{NNN}` | Cross-functional requirement (perf, security, a11y ‚Äî formerly "NFR") | accord, scribe | `CFR-001` |
| `AC-{FEATURE}-{NNN}` | Acceptance criterion (the verifiable unit) | accord (`L3`), attest, scribe | `AC-LOGIN-001` |
| `SC-{AC-id}-{type}-{NNN}` | BDD scenario derived from an AC (`type` ‚àà HP/NP/BP/EP) | attest, accord | `SC-AC-LOGIN-001-HP-001` |
| `IMPL-{NNN}` | Implementation artifact (module/function realizing a REQ/AC) | builder (cited), scribe | `IMPL-001` |
| `TEST-{NNN}` | Test case proving an AC/SC | radar, voyager (cited) | `TEST-001` |
| `DISQ-{NNN}` | Disqualification / hard-fail condition (accord `L4`) | accord | `DISQ-001` |

**Backward-compatibility note.** Legacy `NFR-*` (scribe) maps to `CFR-*`; legacy bare `US`
(accord user story) maps to `REQ-*`. Prefer `CFR`/`REQ` in new packages; accept the legacy forms
on read.

### Linking rule (bidirectional, mandatory)

A requirement is traceable only when the chain resolves **both directions**:

```
REQ-001 ‚îÄ‚ñ∂ AC-LOGIN-001 ‚îÄ‚ñ∂ SC-AC-LOGIN-001-HP-001 ‚îÄ‚ñ∂ TEST-001 ‚îÄ‚ñ∂ IMPL-001
   ‚óÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ every node also links back to its parent ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
```

- Forward gap (`REQ` with no `AC`, `AC` with no `TEST`) = **untested requirement**.
- Backward gap (`TEST`/`IMPL` citing no `AC`) = **orphan artifact** (scope creep signal).
Both are coverage defects, not warnings.

## 2. `.traceability.yaml` (machine-readable ledger)

When a spec package is durable enough to drive implementation (accord `Full`/`Standard`,
scribe SRS, any `nexus acceptance` Tier-S/A run), emit a `.traceability.yaml` at the repo root
(or `docs/`). It is the artifact `attest`, `guardian`, and `judge` read to validate coverage
without re-parsing prose.

```yaml
# .traceability.yaml ‚Äî generated by accord/scribe, verified by attest, gated by guardian/judge
version: 1
feature: <feature-slug>
spec_source: <path-to-spec or pointer>
links:
  - req: REQ-001
    title: <one line>
    priority: CRITICAL | HIGH | MEDIUM | LOW
    acs:
      - id: AC-LOGIN-001
        scenarios: [SC-AC-LOGIN-001-HP-001, SC-AC-LOGIN-001-NP-001]
        tests: [TEST-001]          # filled as tests land; empty = forward gap
        impl: [IMPL-001]           # filled as code lands; empty = unimplemented
        verdict: PASS | PARTIAL | FAIL | NOT_TESTED | AMBIGUOUS   # attest writes this
  - cfr: CFR-001
    title: <one line>
    acs: [AC-PERF-001]
coverage:
  forward: <0.0-1.0>   # ACs with ‚â•1 test / total ACs
  backward: <0.0-1.0>  # tests+impl citing a valid AC / total tests+impl
  orphans: [<ids citing no AC>]
  gaps: [<REQ/AC with no downstream>]
```

### Read/write protocol

| Skill | Role on the ledger |
|-------|--------------------|
| `accord` / `scribe` | **Author** `links[]` (REQ/CFR/AC/SC) at spec time; set initial `verdict: NOT_TESTED`. |
| `attest` | **Verify** ‚Äî fill `verdict`, recompute `coverage`, list `orphans`/`gaps`. The compliance verdict consumes these. |
| `radar` / `voyager` | Cite the `AC-*`/`SC-*` ID in each test (comment header) so `tests[]` can be reconciled. |
| `guardian` / `judge` | **Gate** ‚Äî read `coverage`/`gaps` to block merge when a CRITICAL AC has a forward gap. |

Coverage thresholds align with each skill's existing bars (accord `Full ‚â•95%` / `Standard ‚â•85%`
/ `Lite ‚â•70%`; attest CERTIFIED `‚â•90%`). A missing `.traceability.yaml` is acceptable for `Lite`
/ single-AC work; it is mandatory wherever the authoring skill already requires a traceability
matrix.

---

## 3. Adoption checklist

- New AC ‚Üí use `AC-{FEATURE}-{NNN}`, never a bare number.
- Authoring a durable package ‚Üí emit `.traceability.yaml`; initial verdicts `NOT_TESTED`.
- Writing a test ‚Üí cite the `AC`/`SC` ID it proves in a header comment.
- Verifying ‚Üí `attest` updates verdicts + coverage; never invents IDs not in the ledger.
- Gating ‚Üí `guardian`/`judge` treat a CRITICAL forward gap as a blocking finding.
